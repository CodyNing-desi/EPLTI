# 🔍 EPLTI 竞品深度分析报告

> **项目名称**：EPLTI（英超 TI / Premier League Type Indicator）  
> **定位**：面向英超球迷的 MBTI 式人格测试平台  
> **分析日期**：2026-05-01

---

## 一、项目概述

这是一个模仿 MBTI 人格测试的英超球迷分类网站。用户通过回答 18 道精心设计的题目，被系统分配到 14 种球迷人格中的一种。

### 核心卖点
| 特性 | 说明 |
|------|------|
| 原创维度模型 | T-E-S-K-R（死忠/激情/社交/懂球/韧性）五维模型 |
| 14 种人格类型 | 8 种绑定英超 Big6 球队 + 6 种通用类型 |
| 完整前后端 | React SPA + Node.js/Express + MySQL |
| 社交裂变 | 海报生成 + 微信 JSSDK 分享 + OG 标签 |
| 已部署上线 | 腾讯云 Lighthouse + PM2 |

---

## 二、技术架构评分

### 2.1 前端 ⭐⭐⭐⭐ (4/5)

| 技术选型 | 评价 |
|----------|------|
| React 18 + Vite | ✅ 主流且高效 |
| Tailwind CSS v3 | ✅ 快速搭建 UI，风格统一 |
| Framer Motion | ✅ 动画体验丝滑 |
| TypeScript（部分） | ⚠️ 只有 `data/` 和 `engine/` 用了 TS，页面和组件仍是 JSX |
| html2canvas + QRCode | ✅ 海报生成方案合理 |

**亮点**：
- Quiz 页面的动画切换体验很好（`AnimatePresence` + 自定义方向滑动）
- 进度条用了 `highestIndex` 防止回退时倒流，细节到位
- 结果页内容非常丰富：维度条形图 + SVG 雷达图 + CP 关系 + 统计数据
- 海报生成自动触发（点击即生成 PNG），手机端适配"长按保存"，体验好

**不足**：
- TS 化只做了一半，核心 UI 层（4 个页面 + 2 个组件）全是 `.jsx`，类型安全没贯穿到底
- `utils/` 目录是空的——要么是清理遗留，要么是架构预留没用上
- 没有任何单元测试或 E2E 测试
- 没有路由懒加载（`React.lazy`），所有页面都在首屏打包里
- `ExportAll` 页面放在 `/admin/export` 路由但没有任何权限校验

### 2.2 后端 ⭐⭐⭐ (3/5)

| 技术选型 | 评价 |
|----------|------|
| Express 5 + MySQL | ✅ 简单够用 |
| 手写 Rate Limiter | ⚠️ 仅适用单进程，注释中也说了 |
| 自动建库建表 | ✅ 开箱即用，部署友好 |
| OG 标签动态注入 | ✅ 对社交分享很有价值 |

**亮点**：
- 数据库初始化代码写得规范（先建库后建表，连接池管理清晰）
- Rate limiter 有定期清理过期记录的逻辑，避免内存泄漏
- SPA fallback + OG 注入的 `index.js` 逻辑层次清晰
- VALID_TYPES 白名单校验，防垃圾数据

**不足**：
- **安全性堪忧**：
  - `db.js` 的 `CREATE DATABASE` 语句直接拼接了 `DB_NAME`，存在 SQL 注入风险
  - OG 标签注入时 `typeCode` 直接拼入 HTML，没有转义（XSS 风险）
  - `/admin/export` 前端路由无鉴权
- 微信 JSSDK 路由在无凭据时返回 `mock_ticket`，可能导致前端静默失败
- 没有日志系统（只有 `console.error`）
- 没有数据库迁移工具（直接 `CREATE TABLE IF NOT EXISTS`）

### 2.3 部署 ⭐⭐⭐⭐ (4/5)

- 腾讯云 + PM2 是轻量级部署的标准方案
- 前端 build 后由 Express 静态托管，无需 Nginx 分离
- 但没有 CI/CD 管道，没有自动化部署脚本

## 三、产品设计评分

### 3.1 内容策划 ⭐⭐⭐⭐⭐ (5/5)

> [!TIP]
> 这是整个项目**最强的部分**。

- **18 道题目设计精良**：每道题都指向特定维度，选项分数分布合理
- **14 种人格文案质量极高**：
  - 「冠压抑」（阿森纳）、「弗格森遗老」（曼联）、「受虐狂」（热刺）——命名精准且有传播力
  - 每个人格都有独特的 tagline、完整描述、4 个标签、最佳/最差 CP
  - 文案语气活泼、接地气、带有互联网黑话（PTSD、刮刮乐、速效救心丸），非常适合社交传播
- **维度模型有一定学术底色**：五维模型虽然不是真正的心理学量表，但结构完整，能自圆其说

### 3.2 用户体验 ⭐⭐⭐⭐ (4/5)

| 环节 | 评价 |
|------|------|
| 首页 | ✅ 简洁有力，统计数字增强参与感 |
| 答题 | ✅ 动画丝滑，支持回退修改 |
| 结果 | ✅ 信息密度很高，雷达图+维度条形图+CP+统计 |
| 海报 | ✅ 自动生成 PNG，手机长按保存，传播链路闭合 |
| 分享 | ✅ ID 短链 + OG 标签 + 微信 JSSDK 三管齐下 |

**不足**：
- 首页缺少示例或预览，用户不知道测完是什么样子
- 没有答题结果的历史记录功能
- 移动端没有 PWA 支持

### 3.3 传播设计 ⭐⭐⭐⭐ (4/5)

传播链路：**答题 → 结果页 → 一键海报 → 长按保存/微信分享 → 扫码进入首页 → 新一轮答题**

这个闭环设计得比较完整。特别是：
- 海报底部有二维码指向首页
- 服务端 ID 短链比 Base64 更优雅
- 首页"XXX 人已完成测验"制造从众心理

## 四、算法分析 ⭐⭐⭐⭐ (4/5)

### 4.1 得分机制
```
原始分 → 归一化 (÷ 最大可能值 × 5 ÷ 0.8) → clamp 到 [-5, 5]
```

- `÷ 0.8` 的矫正因子用于撑开分布，对抗"大多数人都会在均值附近"的问题——**这是一个有思考深度的设计**
- `maxPossibleScores` 是运行时从题目数据自动计算的，改题后无需手动同步——**工程化做得好**

### 4.2 主队检测
```
teamHint 积分 + fanType（核心/泛球迷）信号 → 动态阈值 → 决定候选池
```

- 用 `coreFanScore` 调整阈值是一个聪明的设计：越像核心球迷，越容易被检测为 Big6 球队专属类型
- 非 Big6 球迷统一进入通用类型池

### 4.3 匹配算法
- 标准的**欧氏距离**最近邻匹配
- 同时返回次优匹配（`runnerUp`），增加结果丰富度

**不足**：
- 只有 14 种类型，每队最多 2 种，区分度有限
- 没有维度权重（T/E/S/K/R 一视同仁），但实际上"死忠"和"社交"的重要性差异很大
- 没有防作弊机制（比如前后答案矛盾检测）

## 六、综合评价

### 总评：⭐⭐⭐⭐ (4/5) — 完成度很高的 MVP，内容策划是核心竞争力

| 维度 | 评分 |
|------|------|
| 内容策划 | ⭐⭐⭐⭐⭐ |
| 产品设计 | ⭐⭐⭐⭐ |
| 技术实现 | ⭐⭐⭐½ |
| 传播能力 | ⭐⭐⭐⭐ |
| 代码质量 | ⭐⭐⭐ |

### 核心优势（值得学习）
1. **文案质量极高**——14 种人格的命名和描述都有很强的社交传播性
2. **传播闭环完整**——从答题到海报到扫码回流，链路设计成熟
3. **细节打磨到位**——进度条防回退、矫正因子、自动计算 maxScores 等
4. **移动端体验优先**——长按保存图片、safe-area 适配、底部悬浮栏

### 核心弱点（可以针对的方向）
1. **安全漏洞明显**——XSS、SQL 注入风险，`/admin/export` 无鉴权
2. **没有任何测试**——零测试覆盖率
3. **TypeScript 化半途而废**——只有数据和算法层是 TS
4. **类型数量有限**——14 种类型，每队最多 2 种，复测体验单一
5. **没有后台管理**——无法查看用户数据、调整题目、A/B 测试
6. **缺少数据分析**——没有用户行为追踪、转化漏斗
7. **视觉设计偏素**——整体是灰白极简风，缺少品牌感和记忆点

> [!IMPORTANT]
> **如果你要做竞品**，建议在以下方向拉开差距：
> - 更丰富的类型体系（至少 20+ 种）和更深的内容
> - 更强的视觉冲击力和品牌辨识度
> - 加入互动元素（类型对比、好友 PK、赛季更新）
> - 完善的数据分析和后台运营能力
> - 更严格的代码质量和安全标准
